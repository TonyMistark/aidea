# Aidear 下一版本产品需求文档 (PRD)

> 版本: v1.1
> 日期: 2025-07-02
> 状态: ✅ 已确认 — 待实现

---

## 概述

本版本在现有 AI 生成 + Markdown 转换基础上，新增三大核心功能：
1. 直接转换模式支持富文本粘贴（结构性格式保留）
2. 任务队列（后台生成）
3. 主题系统（Theme System）

---

## 需求一：直接转换模式 - 粘贴保留结构性格式

### 1.1 背景与目标

当前「直接转换」模式仅支持纯文本粘贴，用户从网页、Word、微信编辑器复制过来的结构化内容（如加粗、列表、标题）会丢失格式。本功能旨在保留这些结构性格式，让直接转换真正可用。

### 1.2 用户场景

用户从网页或 Word 复制一篇有结构的文章到 Aidear，点击「转换」后，希望保留原有的标题层级、加粗、列表等 Markdown 结构。

### 1.3 功能需求

#### FR-1.1: 剪贴板富文本提取
- **输入**: 用户从剪贴板粘贴的富文本（UTI `public.html` / `public.rtf`）
- **处理**: 从富文本中提取结构性格式
- **输出**: 转换为 Markdown 格式

#### FR-1.2: 保留的格式类型
| 格式类型 | Markdown 语法 | 说明 |
|---------|-------------|------|
| 标题层级 | `# ` / `## ` / `### ` | H1/H2/H3，最多到 H3 |
| 加粗 | `**文本**` | 粗体 |
| 斜体 | `*文本*` | 斜体 |
| 有序列表 | `1. ` / `2. ` | 数字列表 |
| 无序列表 | `- ` / `* ` | 项目符号列表 |
| 引用块 | `> ` | 引用 |
| 链接 | `[文本](URL)` | 超链接 |
| 删除线 | `~~文本~~` | 删除线（可选） |

#### FR-1.3: 降级策略
- 不支持的格式（如字体颜色、背景色、字号、对齐方式等）丢弃为纯文本
- 嵌套列表（多层）降级为单层列表（或用缩进表示）
- 表格降级为纯文本（或 Markdown 表格，视复杂度而定）

#### FR-1.4: UI 交互
- 在「直接转换」模式下，粘贴时自动触发格式提取
- 用户可看到提取后的 Markdown 预览（可选开关）
- 提供「纯文本模式」切换按钮，允许用户关闭格式提取

### 1.4 技术方案

#### 方案 A：使用 NSAttributedString + HTML Parser（推荐）
- 从剪贴板读取 `public.html` / `public.rtf` UTI
- 使用 `NSAttributedString` 解析 HTML → 遍历属性提取 Markdown
- 优点：原生支持，兼容性好
- 缺点：HTML → Markdown 转换需自行实现

#### 方案 B：使用第三方 HTML → Markdown 库
- 如 swift-markdownify 或类似开源库
- 优点：转换逻辑成熟
- 缺点：引入外部依赖，可能体积较大

#### 方案 C：正则 + 简单 DOM 解析
- 手动解析 HTML 标签 → 映射到 Markdown
- 优点：轻量，可控
- 缺点：需要处理各种边界情况

**推荐方案：A（NSAttributedString）或 C（正则解析）** — 先评估 Swift 生态是否有轻量级的 HTML → Markdown 转换工具。

### 1.5 验收标准

- [ ] 从网页复制标题 + 列表 + 加粗文本，粘贴后格式正确
- [ ] 从 Word 复制结构化内容，粘贴后格式正确
- [ ] 不支持的样式（颜色、字号）被丢弃，不影响纯文本内容
- [ ] 纯文本模式开关有效
- [ ] 性能：大文本（>5000 字）粘贴不卡顿（<500ms）

### 1.6 待确认问题

- 是否需要支持从微信编辑器粘贴？（格式较特殊）
- 表格如何处理？直接丢弃还是转为 Markdown 表格？

## 需求二：任务队列（后台生成）

### 2.1 背景与目标

当前点击「开始AI创作」后，用户需等待生成完成才能继续操作。引入任务队列后，用户可以发起多个生成任务，任务在后台执行，用户可继续编辑或发起新任务。

### 2.2 用户场景

1. 用户输入想法 A，点击「开始AI创作」
2. 任务以动画飞到左侧任务列表按钮
3. 用户无需等待，继续输入想法 B，再次点击生成
4. 点击任务列表按钮，查看所有进行中的任务及已完成的结果
5. 任务完成后，可在任务列表中查看结果

### 2.3 功能需求

#### FR-2.1: 任务模型

```swift
struct GenerationTask: Identifiable {
    let id = UUID()
    let inputText: String        // 用户输入的原始文本
    let promptID: UUID?          // 使用的 Prompt（可选）
    let createdAt: Date
    
    var status: TaskStatus       // .pending / .running / .completed / .failed / .cancelled
    var result: GenerationResult? // 生成结果
    var errorMessage: String?    // 错误信息
    var elapsedSeconds: Int      // 已耗时（秒）
    var selectedTheme: Theme?    // 应用的主题（预留）
}

enum TaskStatus: String {
    case pending   = "等待中"
    case running   = "生成中"
    case completed = "已完成"
    case failed    = "失败"
    case cancelled = "已取消"
}
```

#### FR-2.2: 任务管理器（TaskManager）

- 全局单例管理所有任务
- 支持任务状态监听（Observable）
- 支持并发数控制（默认同时执行 1 个，或允许设置）
- 支持任务取消
- 任务结果本地缓存（可选，用于后续查看历史）

#### FR-2.3: UI 交互

**任务列表按钮（悬浮或导航栏）**
- 位置：主界面左上角或导航栏
- 状态：
  - 空：无任务时隐藏或显示为灰色
  - 进行中：显示动画（旋转点 / 进度条），可显示进行中任务数
  - 有结果：显示未读结果数红点

**任务列表弹窗 / 页面**
- 展示所有任务（按时间倒序）
- 任务卡片包含：
  - 输入文本摘要（前 20 字）
  - 状态（等待中 / 生成中 / 已完成 / 失败）
  - 耗时
  - 操作按钮：查看结果 / 重新生成 / 删除
- 正在生成的任务可取消
- 已完成的任务可查看详情（跳转结果页面）

**生成动画**
- 用户点击「开始AI创作」后，按钮区域有一个动画效果
- 动画：一个小的任务卡片缩小并飞到左侧任务列表按钮位置
- 使用 SwiftUI 的 `matchedGeometryEffect` 或自定义动画实现

#### FR-2.4: 主界面调整

- 点击「开始AI创作」后，输入区不清空（允许用户继续输入新内容）
- 结果区不再直接显示在主界面（改为到任务列表中查看）
- 或者：主界面仍可显示最近一次结果，但任务列表可查看所有结果

### 2.4 技术方案

#### 方案 A：纯 SwiftUI + Task 管理（推荐）
- 使用 `@Observable` / `ObservableObject` 管理任务列表
- 每个任务使用 Swift `Task` 异步执行
- 任务管理器控制并发（`TaskGroup` 或自定义队列）

#### 方案 B：引入背景任务框架
- 如使用 `BackgroundTask`（iOS 13+）
- 优点：App 后台也能继续生成
- 缺点：复杂度增加，可能不需要

**推荐方案：A**，当前场景下 App 前台使用已足够。

### 2.5 验收标准

- [ ] 可同时发起多个生成任务
- [ ] 任务列表正确显示所有任务状态
- [ ] 生成动画流畅自然
- [ ] 任务完成/失败后，用户可在任务列表中查看结果
- [ ] 任务取消后状态正确更新
- [ ] 任务列表支持删除已完成/失败的任务

## 需求三：主题系统（Theme System）

### 3.1 背景与目标

当前应用只有一种「微信绿」主题。本功能引入主题系统，允许用户在生成后切换不同主题样式渲染文章，提升文章展示多样性。

主题样式来源：[doocs/md](https://github.com/doocs/md) — 一个流行的 Markdown 排版工具，提供多种精美的公众号文章排版主题。我们从中提取 `default`、`grace`、`simple` 三个主题适配到 Aidear 中。

### 3.2 用户场景

1. 用户生成文章后，在结果页面看到当前主题（微信绿）
2. 用户点击「切换主题」按钮，弹出主题列表
3. 用户选择另一个主题（如「知乎蓝」、「深色模式」），文章立即重新渲染
4. 用户可预览主题效果，确认后应用到当前文章

### 3.3 功能需求

#### FR-3.1: 主题模型

```swift
struct Theme: Identifiable, Codable, Equatable {
    let id = UUID()
    let name: String           // 主题名称（如「微信绿」、「知乎蓝」）
    let identifier: String     // 唯一标识（如 "wechat-green"）
    let previewImage: String?  // 预览图名称（可选）
    
    // CSS 样式（Markdown → HTML 渲染时注入）
    let cssStyles: ThemeStyles
    
    struct ThemeStyles: Codable {
        // 全局
        let fontFamily: String
        let fontSize: String
        let lineHeight: String
        let textColor: String
        let backgroundColor: String
        
        // 标题
        let headingColor: String
        let headingFontSize: String
        let headingFontWeight: String
        
        // 引用块
        let blockquoteBorderColor: String
        let blockquoteBackgroundColor: String
        
        // 链接
        let linkColor: String
        
        // 列表
        let listMarkerColor: String
        
        // 代码块
        let codeBackgroundColor: String
        let codeTextColor: String
        
        // 其他...
    }
}
```

#### FR-3.2: 主题管理器（ThemeManager）

- 全局管理所有可用主题
- 提供默认主题列表（内置）
- 支持主题切换（实时预览）
- 持久化用户选择的主题（按任务或全局）

#### FR-3.3: UI 交互

**主题切换按钮**
- 位置：结果页面的操作按钮区域
- 图标：调色板 / 主题图标
- 点击：弹出主题选择弹窗

**主题选择弹窗**
- 网格或列表展示所有主题
- 每个主题显示：预览图 + 名称
- 当前主题高亮标记
- 点击即切换预览，点击「应用」确认

**内置主题（第一期）**

| 主题名称 | 标识 | 来源 | 说明 |
|---------|------|------|------|
| 微信绿 | wechat-green | Aidear 原生 | 当前默认主题，绿色调，适合公众号 |
| 经典 | doocs-default | doocs/md | 最完整的排版主题，覆盖所有元素 |
| 优雅 | doocs-grace | doocs/md | 圆角阴影优雅风格 |
| 简洁 | doocs-simple | doocs/md | 简洁现代风格 |

#### FR-3.4: Markdown → HTML 渲染调整

- `MarkdownWebView` 需要接受动态 CSS 参数
- 主题切换时，重新加载 HTML 并注入新的 CSS 样式
- 保持原始 Markdown 不变，仅改变渲染样式

从 [doocs/md](https://github.com/doocs/md) 项目提取了以下主题样式：

| 主题名称 | 标识 | 来源 | 特点 |
|---------|------|------|------|
| 经典 (default) | `default` | doocs/md 默认主题 | 最完整的主题，包含 GFM Alert、学术环境、表格、公式等 |
| 优雅 (grace) | `grace` | @brzhang | 优雅的视觉效果，圆角、阴影、渐变分割线 |
| 简洁 (simple) | `simple` | @okooo5km | 简洁现代，最小化装饰，适合技术文档 |

#### FR-3.5: 主题样式来源与适配

**来源项目**: [doocs/md](https://github.com/doocs/md) - 一个流行的 Markdown 排版工具，提供微信公众号、知乎等平台的文章排版。

**样式结构**:
```
base.css      → 基础容器样式（字体、行高、margin reset）
default.css   → 经典主题（最完整，覆盖所有元素）
grace.css     → 优雅主题（基于 default，覆盖部分样式）
simple.css    → 简洁主题（基于 default，覆盖部分样式）
```

**CSS 变量定义**（每个主题需定义）：
| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `--md-primary-color` | 主题主色 | `#07c160` (微信绿) |
| `--md-font-family` | 正文字体 | `-apple-system, ...` |
| `--md-font-size` | 正文字号 | `14px` |
| `--foreground` | 前景色 HSL | `0 0% 9%` |
| `--blockquote-background` | 引用块背景 | `rgba(0,0,0,0.03)` |

**CSS 文件已保存到项目**（实现时直接使用）：
```
docs/superpowers/themes/
├── base.css                → 基础容器样式（52 行）
├── default.css             → 经典主题（563 行，最完整）
├── grace.css               → 优雅主题（136 行）
├── simple.css              → 简洁主题（130 行）
└── default-custom-theme.txt → 自定义主题模板（284 行，供未来新增主题参考）
```

**适配要点**：
1. 从 doocs/md 提取的 CSS 是面向 Web 的，需要适配 iOS WebView 环境
2. 部分 CSS 特性（如 `color-mix()`）在 iOS WebView 中可能需要 fallback
3. 移除与 Aidear 无关的样式（如 KaTeX 公式、VSCode 相关）
4. 主题切换时，注入对应主题的 CSS 到 WebView
5. 主题采用叠加机制：`base.css` + 主题专属 CSS（如 `default.css`、`grace.css`）

### 3.4 技术方案

#### 方案 A：CSS 变量动态注入（推荐）
- HTML 模板中使用 CSS 变量（`var(--theme-primary)`）
- 主题切换时，动态替换 CSS 变量值
- 优点：切换流畅，无需重新解析 Markdown
- 缺点：需要重构现有 HTML 生成逻辑

#### 方案 B：完整 CSS 覆盖
- 每个主题提供完整的 CSS 字符串
- 切换时替换整个 `<style>` 标签
- 优点：实现简单，每个主题完全独立
- 缺点：每次切换需要重新渲染 WebView

**推荐方案：B（完整 CSS 覆盖）**，先保证功能可用，后续优化性能。

### 3.5 验收标准

- [ ] 结果页面显示「切换主题」按钮
- [ ] 点击弹出主题列表（目前只有「微信绿」）
- [ ] 主题切换后文章样式正确更新
- [ ] 主题选择持久化（切换 App 后保持）
- [ ] 新增主题时只需修改配置文件即可生效
