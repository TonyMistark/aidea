# Aidear AI 引擎设计文档

---

## 1. 设计目标

AI 引擎是 Aidear 的核心竞争力。它需要做到：

1. **理解碎片化输入**：用户可能只说关键词、短句、零散想法，AI 需要推断意图并补全
2. **生成高质量文章**：产出符合公众号阅读习惯、结构清晰、可读性强的中文文章
3. **风格可控**：支持不同的文章类型、语气风格、目标长度
4. **流式体验**：边生成边展示，减少等待感
5. **安全可靠**：妥善处理错误、内容过滤、Token 限制

## 2. 多服务商支持

### 2.1 服务商抽象

```
AIServiceProtocol
    │
    ├── OpenAIService      (chat/completions, SSE streaming)
    ├── AnthropicService   (messages, SSE streaming)
    └── CustomOpenAIService (OpenAI-compatible endpoint)
```

### 2.2 服务商选择逻辑

```swift
enum AIProvider: String, Codable, CaseIterable {
    case openAI
    case anthropic
    case custom
}

// 工厂方法
struct AIServiceFactory {
    static func create(
        provider: AIProvider,
        apiKey: String,
        model: String,
        baseURL: String?
    ) -> AIServiceProtocol {
        switch provider {
        case .openAI:
            return OpenAIService(apiKey: apiKey, model: model)
        case .anthropic:
            return AnthropicService(apiKey: apiKey, model: model)
        case .custom:
            return CustomOpenAIService(
                apiKey: apiKey,
                model: model,
                baseURL: baseURL ?? ""
            )
        }
    }
}
```

### 2.3 各服务商默认模型

| 服务商 | 默认模型 | 上下文窗口 | 备注 |
|--------|----------|------------|------|
| OpenAI | `gpt-4o` | 128K | 性价比最高 |
| Anthropic | `claude-sonnet-4-6` | 200K | 中文写作出色 |
| 自定义 | 用户指定 | 取决于服务 | 兼容 OpenAI API 格式 |

## 3. System Prompt 设计

### 3.1 基础 System Prompt（通用部分）

```
你是一位专业的公众号文章写手，擅长将碎片化的想法和观点整理成高质量的中文文章。

你的写作原则：
- 尊重用户的原始观点和意图，不擅自改变核心意思
- 逻辑清晰，结构合理，段落之间有自然的过渡
- 语言简洁有力，避免空洞的套话和过度修饰
- 符合微信公众号的阅读习惯：段落短（3-5句）、善用小标题、金句突出
- 标题要吸引人但不标题党，准确反映文章内容

输出格式（Markdown）：

# [文章标题]

> [引语/摘要，1-2句话概括文章核心观点]

---

[正文内容，使用 Markdown 格式组织]

## [小标题]

[段落内容]

> [如需引用，使用引用格式]

## [小标题]

[段落内容]

---

[结尾段落，总结或升华]

---
[备用标题1]
[备用标题2]
```

### 3.2 按文章类型的 Prompt 追加

#### 观点文
```
这是一篇观点文。要求：
- 开头直接亮出核心观点
- 提供 2-3 个支撑论据，每个论据有案例或类比
- 论证有层次感，从现象到本质
- 结尾呼应开头，升华观点
- 语气：有态度但不偏激
```

#### 干货文
```
这是一篇干货文。要求：
- 开头说明读者将获得什么
- 知识点分点清晰，使用数字序号
- 每个知识点包含"是什么 - 为什么 - 怎么做"
- 善用类比帮助理解
- 结尾有可操作的总结/清单
```

#### 故事文
```
这是一篇故事文。要求：
- 有明确的场景感和画面感
- 以故事开头，带入读者情绪
- 故事中暗含观点（不要直接说教）
- 结尾从故事中提炼感悟
- 语言有温度，像朋友在讲述
```

#### 资讯文
```
这是一篇资讯文。要求：
- 5W1H 完整：何时、何地、何人、何事、为何、如何
- 信息密度高，不啰嗦
- 客观中立，事实与评论分开
- 结构：导语 → 正文 → 背景/展望
```

#### 自由风格
```
请根据用户的输入内容，自主判断最合适的文章类型和写作风格。
```

### 3.3 按语气的 Prompt 追加

| 语气 | Prompt 指令 |
|------|------------|
| 正式专业 | "使用正式专业的语言，保持严谨客观。用词准确规范，可适当使用行业术语。" |
| 轻松亲切 | "使用轻松亲切的语气，像和好朋友聊天一样。多用'你'拉近距离，语言自然口语化。" |
| 犀利有料 | "语言犀利有力，一针见血。观点鲜明不模糊，敢于直击要害。可以适当使用反问和对比。" |
| 温暖治愈 | "用温柔的文字表达，给人力量和安慰。多关注人性和情感，让读者感受到理解和接纳。" |

### 3.4 按目标长度的 Prompt 追加

| 长度 | Prompt 指令 |
|------|------------|
| 短文 (~800字) | "文章控制在 800 字左右。开门见山，快速切入主题。每个观点简洁表达，不做过多展开。" |
| 中篇 (~1500字) | "文章控制在 1500 字左右。每个观点有适当的展开和例证。结构完整但不过于冗长。" |
| 长文 (~2500字) | "文章控制在 2500 字左右。深度剖析主题，每个观点充分展开。可以使用更多案例和数据。" |

### 3.5 按读者对象的 Prompt 追加

| 读者 | Prompt 指令 |
|------|------------|
| 普通读者 | "面向普通读者，语言通俗易懂，避免专业术语。用生活化的例子帮助理解。" |
| 行业人士 | "面向行业人士，可以适当使用专业术语和行业背景知识，不必过多解释基础概念。" |
| 年轻群体 | "面向年轻读者，语言可以新潮活泼，适当使用网络流行语和年轻化的表达。" |

## 4. 对话式调整 Prompt

### 4.1 System Prompt（调整模式）

```
你正在帮助用户修改一篇已经生成的文章。用户会给出具体的修改指令。

当前文章：
---
{articleContent}
---

规则：
- 只修改用户指定的部分，其余内容保持不变
- 修改后返回完整的文章内容（Markdown 格式）
- 在修改完成后，用 1-2 句话说明你做了什么修改
- 如果用户的指令不明确，先确认再修改

回复格式：
【修改说明】
[1-2句话说明修改了什么]

【更新后的文章】
[完整的修改后文章，Markdown 格式]
```

### 4.2 快捷指令映射

| 快捷指令 | 实际 Prompt |
|----------|------------|
| "改得更吸引人" | "请优化文章开头和标题，使其更有吸引力。开头可以用悬念、故事、问题或对比来抓住读者注意力。" |
| "精简一点" | "请精简文章内容，删除冗余表达，合并相似观点，让文章更加紧凑有力。目标减少约 30% 篇幅，保留核心观点。" |
| "增加一个案例" | "请在文章合适的位置增加一个具体的案例或故事来支撑观点，让内容更加有说服力和可读性。" |
| "调整语气" | "请将文章的语气调整为更温和/更犀利的风格。"（需要进一步确认方向） |

## 5. Token 管理

### 5.1 Token 预算分配

文章生成请求的 Token 分配策略：

```
总上下文窗口：128K (GPT-4o) / 200K (Claude)

System Prompt：    ~2K tokens     (固定)
用户输入：          ~1-4K tokens   (取决于用户说了多少)
输出预留：          ~4K tokens     (2500字中文 ≈ 4000 tokens)
─────────────────────────────────
剩余作为缓冲：      ~118K tokens   (GPT-4o)
```

### 5.2 输入长度检查

| 输入长度 | 行为 |
|----------|------|
| < 100 字 | 正常处理，AI 会扩展 |
| 100-3000 字 | 最佳范围 |
| 3000-5000 字 | 正常处理，提示"可能需要更长时间" |
| 5000-10000 字 | 截断到 5000 字（取前面部分），提示用户 |
| > 10000 字 | 拒绝生成，提示"内容过长，请精简" |

### 5.3 成本估算（以 OpenAI GPT-4o 为例）

| 操作 | 输入 Token | 输出 Token | 估算成本 |
|------|------------|------------|----------|
| 生成短文 | ~2,500 | ~1,200 | ~$0.02 |
| 生成中篇 | ~3,000 | ~2,200 | ~$0.03 |
| 生成长文 | ~3,500 | ~3,500 | ~$0.05 |
| 对话调整 | ~5,000 | ~3,000 | ~$0.04 |

*注：用户自带 API Key，费用由用户自行承担。*

## 6. 响应解析

### 6.1 文章解析规则

AI 返回 Markdown 文本，需解析为结构化文章：

```
返回内容（Markdown）
    │
    ├─ 提取 # 标题       → Article.title
    ├─ 提取 > 引语       → Article.subtitle
    ├─ 提取正文           → Article.content (Markdown)
    ├─ 统计字数           → Article.wordCount
    └─ 提取备用标题       → Article.alternativeTitles
        (正文末尾独立的 [备用标题X] 行)
```

### 6.2 解析失败处理

如果 AI 返回内容不符合预期格式：

1. 尝试宽松匹配（找第一个 `#` 行作为标题）
2. 如果完全没有标题 → 提取文章前 30 字作为标题
3. 如果无法解析 → 将全部内容作为正文展示，用户可手动编辑
4. 备用标题缺失 → 不展示标题选择器，只有主标题

### 6.3 内容安全过滤

- 不主动过滤 AI 输出（AI 服务商自身有内容策略）
- 但如果解析到明显的不安全内容（如 Markdown 中包含可执行脚本），做 sanitize 处理
- 不存储或传输用户文章到 Aidear 自己的服务器

## 7. 流式输出体验设计

### 7.1 流式解析

AI API 返回 SSE 流，逐 chunk 解析：

```
raw SSE stream
    │
    ├─ 过滤非 data: 行
    ├─ 解析 JSON chunk
    ├─ 提取 delta.content
    ├─ 累积到 buffer
    └─ 实时推送到 UI
          │
          └─ UI 逐字/逐句展示（带淡入动画）
```

### 7.2 展示策略

| 策略 | 说明 |
|------|------|
| **逐句展示** | 不是逐字，而是累积到完整句子（遇到句号/换行/30字）后展示，避免碎片感 |
| **淡入动画** | 新出现的文字有 0.2s 淡入过渡 |
| **标题醒目** | 第一句标题出现时稍微放大并高亮 |
| **进度提示** | 即使流式展示，上面的阶段文案仍然切换 |

### 7.3 流式中断恢复

```
流式连接中断
    │
    ├─ 已收到内容 < 100 字 → 无法使用，提示重新生成
    ├─ 已收到内容 ≥ 100 字 → 保留已生成内容
    │     提示："生成过程中连接中断，已保留生成的部分内容。"
    │     操作：[重试完整生成] [继续编辑现有内容]
    └─ 网络恢复 → 可以基于已有内容让 AI 继续补全
```

## 8. Prompt 版本管理

### 8.1 版本策略

- Prompt 存储在 UserDefaults，用户可在设置页查看和编辑
- 内置默认 Prompt 作为初始值（`AppSettings.defaultPrompt`）
- 修改保存后即时生效，影响所有新生成的文章
- 提供「恢复默认」按钮，一键还原到内置版本
- 后续可通过远程配置更新 Prompt（P2 考虑）

### 8.2 Prompt 测试

发布前对每种类型 + 语气的组合进行 Prompt 测试：
- 使用固定测试输入
- 检查输出格式是否符合要求
- 检查文章质量（人工评审）
- 记录 Token 消耗

## 9. 未来扩展

### 9.1 多阶段生成（P2）

当前是单次 API 调用生成全文。未来可优化为多阶段：

```
阶段 1：理解意图 + 生成大纲
    │
阶段 2：根据大纲逐段生成正文
    │
阶段 3：生成标题（基于完整正文）
    │
阶段 4：润色 + 格式优化
```

好处：每阶段可控、可干预、质量更高。
代价：API 调用次数 × 4，成本和时间都会增加。

### 9.2 用户自定义 Prompt 模板（✅ 已实现）

- 允许高级用户编辑 System Prompt — 在设置页提供 TextEditor 编辑区
- 修改保存后即时生效，覆盖所有新生成的文章
- 提供「恢复默认」按钮，支持一键还原
- 当前持久化到 UserDefaults，后续可扩展到 iCloud 同步

#### 未来方向（P2）
- 提供 Prompt 模板市场（社区共享）
- 支持多套 Prompt 预设切换
- 需要安全审查机制

### 9.3 文章配图生成（P2）

- 调用 DALL-E / 其他图片生成 API
- 为文章生成封面图或配图
- 自动插入到文章合适位置
